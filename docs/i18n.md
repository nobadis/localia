# Internacionalización (i18n) — Versión en inglés

El frontend está preparado para una versión totalmente en inglés.

## Cómo funciona

- **Locale por defecto:** `es` (español).
- **Locale inglés:** `en`. Se activa con la variable de entorno `NEXT_PUBLIC_LOCALE=en`.

## Archivos

| Ruta | Descripción |
|------|-------------|
| `src/lib/i18n.ts` | `getLocale()`, `getMessages(locale)`, `getLanding(locale)` |
| `src/content/messages/es.ts` | Mensajes de navegación, footer y común (ES) |
| `src/content/messages/en.ts` | Mensajes de navegación, footer y común (EN) |
| `src/content/landing.ts` | Contenido de la landing (ES) — fuente actual |
| `src/content/landing-en.ts` | Contenido de la landing (EN) |

## Cómo ver la web en inglés

1. En `frontend/.env.local` (o `.env`):
   ```
   NEXT_PUBLIC_LOCALE=en
   ```
2. Reiniciar el servidor de desarrollo (`npm run dev` o `pnpm dev`).
3. Reconstruir si es producción: `NEXT_PUBLIC_LOCALE=en pnpm build`.

La cabecera usa `getLocale()` (servidor y cliente) y el layout pone `lang="en"` y el script `window.__LOCALE__` para que el cliente use el mismo idioma. El **footer** ya usa `getMessages(getLocale()).footer.madeWith` (es/en).

## Próximos pasos para 100 % inglés

Para que toda la landing y el resto de la UI estén en inglés cuando `NEXT_PUBLIC_LOCALE=en`:

1. **Landing (page.tsx):** En lugar de importar desde `@/content/landing`, usar `getLanding(getLocale())` y tomar de ahí `HERO`, `PRICING_TIERS`, etc.
2. **Navbar:** Usar `getMessages(getLocale()).nav` para las etiquetas de enlaces y `getLanding(getLocale())` para `DOCS`, `GUIA_IA_PRIVADA`, `CONTACT_EMAIL`, `DEMO_SUBJECT`.
3. **Metadata (layout):** Hacer `metadata` dinámico en función de `getLocale()` (p. ej. `generateMetadata`) o definir dos objetos (es/en) y elegir por locale.
4. **Demo y docs:** Si tienen copy fija, moverla a `messages` o a contenido por locale y usar `getMessages(getLocale())` o el equivalente.

Con eso, un único build con `NEXT_PUBLIC_LOCALE=en` sirve la versión en inglés completa.
